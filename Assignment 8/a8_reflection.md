# a8_reflection.md - Reflection on State and Activity Modeling
## SentinelPay: Real-Time Fraud Detection & Prevention Engine

> **Assignment 8 - Reflection**
> Author: Teboho Nkosi | April 2026

---

The central challenge in this assignment was granularity - deciding how much detail to include in each diagram without making it unreadable. The feedback from Assignment 5 ("use case diagrams are difficult to read") was directly in my mind throughout this work. I made a deliberate decision to give each object and each workflow its own separate diagram rather than combining multiple concerns into one. Eight state diagrams and eight activity diagrams totalling sixteen separate Mermaid blocks is more files to manage but each diagram is focused and readable in isolation.

The granularity problem was sharpest for the state diagrams. The Transaction object could theoretically have twenty states if you modelled every intermediate Kafka offset commit, every Redis write, every PostgreSQL transaction. But that level of detail belongs in sequence diagrams, not state diagrams. The question I kept asking was: from the perspective of a business stakeholder reading this diagram, does this state represent a meaningfully different condition for this object? Received and Enriched are meaningfully different - one has raw PII, the other has tokenised fields. Ingested and Consumed-by-Feature-Engineering-Service are not meaningfully different from a business perspective. That framing - business-meaningful condition changes - helped me cut the Transaction diagram down to eight states from an initial draft of fifteen.

The alignment between state diagrams and Agile user stories surfaced a tension I had not anticipated. User stories are written from a user perspective: "As a Bank Customer, I want to receive a notification when a transaction is blocked." But state diagrams are written from an object perspective: the Transaction object transitions from DecisionPending to HardBlocked. These two perspectives describe the same system behaviour but from opposite angles. Mapping between them required me to think about which user story drove each state transition - the Transaction's HardBlocked state is driven by US-004 (ML scoring), US-006 (notification is a consequence), and US-011 (audit is a parallel consequence). One state transition can satisfy multiple user stories simultaneously.

The comparison between state diagrams and activity diagrams is worth being explicit about. A state diagram answers: "what are all the conditions an object can be in and what events change it?" An activity diagram answers: "what is the sequence of actions that achieves a goal, and who is responsible for each?" They are complementary, not redundant. The Transaction object's state diagram shows that a Transaction can be Approved, SoftDeclined, or HardBlocked - but it does not explain how the notification gets to the customer. The Fraud Decision Enforcement activity diagram shows exactly how: the Decision Engine publishes the decision event, the Notification Orchestrator consumes it in parallel with the Audit Service, and the notification reaches the customer through the external gateway. One diagram cannot replace the other.

The activity diagrams were more difficult to keep readable than the state diagrams because they involve multiple actors in swimlanes. The key lesson from Assignment 5's feedback is that when too many elements are crammed into one visual space, the diagram loses its value as a communication tool. I kept each activity diagram to one specific workflow and used swimlanes strictly by actor - a swimlane only contains actions that actor is responsible for, never actions from two different actors blended together. Parallel actions were represented honestly: where the system genuinely does two things at once (three ML models running concurrently, notification and audit consuming the same Kafka event simultaneously), the diagram shows a fork. Where the system is sequential, no artificial parallelism was added.

The real-time nature of SentinelPay made the guard conditions in the state diagrams particularly important. The 100ms end-to-end latency SLA from NFR-P1 means that every state transition in the Transaction lifecycle must complete within a fraction of that budget. Modelling the circuit breaker state in the Kafka Consumer Group Offset object, and the fallback scoring path in the ML Scoring workflow, reflects the system's designed resilience: the state machine never gets stuck in a state with no exit path, and every failure state has a defined transition that keeps the system operational.
